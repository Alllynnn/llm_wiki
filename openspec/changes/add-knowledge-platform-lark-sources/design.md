## 背景

当前应用已经是一个可工作的单项目 LLM Wiki 引擎。一个项目包含 `purpose.md`、`schema.md`、`raw/sources`、`wiki`、资料导入、自动知识提取、搜索、图谱、聊天和最近项目持久化。现在的产品方向是把它升级成可复用的业务知识库平台，其中黑客松项目只是第一个自用试点项目，而不是一次性特例。

第一阶段的数据源层刻意收窄并外置：飞书/Lark 导出由 Codex skill 负责，该 skill 使用本地已有的 Lark CLI 能力。LLM Wiki 应用在本变更中不执行 Lark CLI。钉钉、飞连、直接飞书 OpenAPI 同步、定时同步、通用连接器管理均不在本次范围内。

## 目标 / 非目标

**目标：**

- 在现有 LLM Wiki 项目之上增加平台目录层。
- 持久化项目分类、项目类型、业务上下文、资料策略、试点/自用试点元数据。
- 用业务化 `purpose.md` 和 `schema.md` 初始化新项目，让现有知识提取能生成更有业务价值的知识。
- 定义未来 Lark CLI Codex skill 写入项目本地文件与元数据的交接契约。
- 当 skill 产出的 Lark 文件出现在 `raw/sources` 下时，复用现有资料监控器、资料生命周期逻辑和知识提取队列。
- 保持现有单项目工作流兼容。

**非目标：**

- 不重写知识提取管线。
- 不创建通用连接器框架。
- 不实现应用内 Lark 导入器。
- 不在 LLM Wiki 应用中解析或执行 Lark CLI。
- 不实现钉钉、飞连、直接飞书 OpenAPI、webhook 或定时同步。
- 不实现跨项目搜索或公司级知识汇总。
- 不在核心 wiki 行为里增加黑客松专用分支。

## 决策

### 决策 1：平台目录是项目之上的元数据层，不是新的存储模型

项目分类和项目类型应表示为普通 LLM Wiki 项目的元数据。现有项目结构保持不变：

```text
project/
  .llm-wiki/
    project.json
    sources/
      lark-sources.jsonl
  purpose.md
  schema.md
  raw/
    sources/
      lark/
  wiki/
```

理由：

- 现有项目模型已经能自然表达“一个真实业务项目 = 一个知识库”。
- 现有打开、创建、最近项目、搜索、图谱、聊天行为都可以继续复用。
- 缺失元数据的老项目可以在读取时归一化到 `general` 分类，保证兼容。

考虑过的替代方案：

- 新建平台数据库：能力更强，但第一阶段没有必要，也会重复项目身份和持久化逻辑。
- 只用文件夹名编码分类：简单，但项目移动或分类调整后很脆弱。

### 决策 2：用业务模板初始化 purpose/schema，而不是改知识提取逻辑

创建项目时选择分类和项目类型，然后生成描述业务知识结构的 `purpose.md` 与 `schema.md`。现有知识提取管线继续读取这些文件并生成 wiki 页面。

理由：

- 当前知识提取已经使用项目 purpose/schema 作为领域指令。
- 业务效果可以通过模板提升，而不是在知识提取逻辑里加很多条件分支。
- 模板在自用试点过程中更容易测试和迭代。

考虑过的替代方案：

- 在知识提取逻辑中硬编码标注/爬虫/自用试点分支：短期更快，但会让引擎不够通用。
- 为每种项目类型创建独立知识提取管线：过早抽象，验证成本高。

### 决策 3：Lark CLI 导出归 Codex skill，不归应用

未来的 Codex skill 使用 Lark CLI 从飞书/Lark 导出项目资料。LLM Wiki 在本变更中的职责是定义该 skill 写入的交接契约：

1. 资料文件写入 `raw/sources/lark/<resource-slug>/`。
2. 来源记录写入 `.llm-wiki/sources/lark-sources.jsonl`。
3. 资料文件使用 Markdown、纯文本、JSON 或导出文档等现有可知识提取格式。
4. 现有资料监控器/导入逻辑把这些文件视作普通项目资料。
5. 当 LLM 配置可用时，现有知识提取队列处理这些文件。

LLM Wiki 应用不直接调用飞书 OpenAPI，也不运行 Lark CLI。凭证、租户授权、CLI 命令细节、分页、导出机制都属于 Codex skill 的职责。

理由：

- 符合当前“只使用 Lark CLI 能力”的范围。
- 在平台 MVP 验证前，把飞书导出机制留在产品外部，避免过早做连接器生命周期。
- 让应用保持为通用知识库平台，而不是连接器宿主。
- Codex skill 可以快速演进，不需要等待应用发版。
- 仍然为后续直接连接器保留来源追踪信息和项目本地原始资料。

考虑过的替代方案：

- 现在做应用内 Lark 导入器：UI 上方便，但会过早引入 CLI 执行、凭证、错误上报、连接器生命周期工作。
- 现在直接接飞书 OpenAPI：能力更完整，但过早引入凭证、权限、限流、租户处理。
- 只做手动复制粘贴：最快，但缺少可重复导出流程和来源追踪约定。

### 决策 4：skill 产出的 Lark 资料使用项目本地来源清单

每次 skill 产出 Lark 导出资料时，记录如下元数据：

- `channel`: `lark`
- 原始 Lark URL 或标识符（如可用）
- 导出/导入时间
- 导出工具名称/版本（如可用）
- 项目相对资料路径
- 内容 hash
- 导出标题/类型（如可用）

来源清单使用项目相对路径，保证项目文件夹移动后仍可用。对于 Markdown 输出，可以在 frontmatter 中同步关键来源字段，但来源清单是事实来源，因为 skill 后续可能导出 PDF、图片或混合资产。

理由：

- `raw/sources` 应继续作为持久原始资料位置。
- 来源清单不依赖 Markdown frontmatter，能覆盖 PDF、图片或混合导出。
- 来源追踪信息支持后续重新导入、去重、审计和资料删除行为。

考虑过的替代方案：

- 只把元数据写进 Markdown frontmatter：无法覆盖非 Markdown 导出。
- 只把元数据写进用户级配置：项目文件夹分享后会丢失来源信息。

### 决策 5：黑客松是普通自用试点项目类型

当前黑客松项目应建模为 `dogfooding` 分类下的普通项目，项目类型可以是 `hackathon-pilot` 或 `ai-knowledge-base-pilot`。

理由：

- 用真实项目压力验证平台。
- 避免做无法复用的一次性路由、模板或导入规则。
- 后续自用试点项目可以沿用同一模型。

## 风险 / 权衡

- [风险] 未来 Codex skill 写出的文件形态可能不适合应用知识提取。-> 缓解：现在定义很小的交接契约：`raw/sources/lark/...` 文件加 `.llm-wiki/sources/lark-sources.jsonl` 来源追踪记录。
- [风险] 业务分类体系可能过于公司定制。-> 缓解：分类 ID 保持足够通用，展示文案和模板可以独立演进。
- [风险] 老项目缺少元数据。-> 缓解：读取时归一化缺失元数据，只在用户保存/更新项目时再持久化默认值。
- [风险] 大型 skill 导出可能压垮知识提取流程。-> 缓解：复用资料监控的文件大小/类型规则和现有知识提取队列。
- [风险] 如果应用执行 CLI，凭证可能进入应用日志。-> 缓解：本变更中应用不执行 Lark CLI，CLI 授权留在 Codex skill 环境。

## 迁移计划

1. 现有项目继续正常打开。
2. 项目 list/open 时，缺失业务元数据的项目按 `general` 处理。
3. 通过平台目录新建的项目写入业务元数据和初始化模板。
4. 现有 `raw/sources` 文件保持不变。
5. 未来 Codex skill 导出时，在 `raw/sources/lark/` 下新增文件，并追加项目本地 Lark 来源清单记录。
6. 回滚时可以忽略元数据和 Lark 来源清单，因为核心项目文件仍与现有单项目引擎兼容。
