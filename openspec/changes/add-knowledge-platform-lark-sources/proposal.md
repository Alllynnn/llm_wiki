## 为什么

LLM Wiki 现在已经具备单项目知识库引擎能力，但黑客松自用试点场景需要它进一步成为一个可复用的业务知识库平台：用户先按业务分类选择知识库，再让项目资料持续沉淀成结构化 wiki 知识。

第一阶段的数据源层不应该在应用内做连接器框架。飞书/Lark 导出由单独的 Codex skill 负责，该 skill 使用现有 Lark CLI 能力导出资料，再把结果作为普通项目资料交给 LLM Wiki。

## 变更内容

- 增加平台首页/知识库目录体验，按业务知识分类组织 LLM Wiki 项目，而不是只展示最近打开的本地项目。
- 增加项目元数据：分类、项目类型、业务上下文、资料策略、自用试点/试点标记。
- 增加面向业务的项目模板：标注项目、爬虫项目、自用试点项目、项目资产等。
- 根据选择的业务分类和项目类型生成 `purpose.md` 与 `schema.md`，让现有知识提取产出更贴合业务的 wiki 页面。
- 定义 Lark CLI 资料交接契约，供后续 Codex skill 使用：飞书/Lark 导出资料写入项目本地 `raw/sources` 路径，并保留可追溯元数据。
- Codex skill 写入文件后，复用现有资料监控、导入和知识提取行为；本变更不在 LLM Wiki 应用内增加 Lark 导入器。
- 明确暂缓钉钉、飞连、直接飞书 OpenAPI 同步、定时同步、跨项目搜索。

## 能力范围

### 新增能力

- `knowledge-platform-catalog`：业务分类目录、项目元数据、项目类型选择，以及面向通用 LLM Wiki 平台的模板化项目创建。
- `lark-source-ingestion`：定义由 Codex skill 产出的飞书/Lark 导出资料如何作为可追溯原始资料进入项目，并复用现有 LLM Wiki 知识提取行为。

### 修改能力

- 无。

## 影响范围

- 前端项目入口：欢迎页/首页、创建项目弹窗、项目列表分组、中英文 UI 文案。
- 项目元数据存储：项目本地 `.llm-wiki` 身份/元数据，以及用户级最近项目持久化。
- 项目模板：业务化 `purpose.md` 与 `schema.md` 初始化。
- 资料生命周期约定：资料元数据、原始资料文件夹约定，以及现有监控器/队列对 skill 产出文件的处理。
- 后端/API 可能需要为 list/open/create 流程暴露项目元数据。
- 外部运行时依赖归属于未来的 Codex skill，而不是 LLM Wiki 应用：本变更中应用不需要解析或执行 Lark CLI。
